# 04 定时任务规格

> 对应数据表见 01-数据库设计.md；触发的消息见 05-消息模板与变量规范.md

## 1. 机器人健康检测（建议每小时一次，或更高频）

遍历 `bots` 表所有记录，逐一调用 Telegram `getMe` 接口：

- 调用成功：更新 `last_health_check_time`，若原状态为异常则恢复为启用
- 调用失败：将该记录 `status` 置为"异常"；若失败的正是 `is_active=true` 的当前生效机器人，立即触发管理员告警（站内告警 + 可选邮件/其他即时通知渠道），提示需要人工切换备用机器人

## 2. 域名健康检测（建议每小时一次）

遍历 `domains` 表所有记录，对每个域名发起可访问性检测（如请求其健康检查路径），根据结果更新 `status` 与 `last_check_time`/`last_check_result`。异常域名自动从"短链生成"的可分配池（status=启用的集合）中排除，不做告警强制要求，但建议在数据面板可见。

## 3. 会员活跃度层级更新（每日一次，建议凌晨低峰期执行）

遍历 `users` 表（可分批处理避免锁表），基于 `now - last_active_time` 的天数差，按 `points_config.activity_tag_thresholds` 配置的阈值（默认：活跃≤7天，待唤醒8-30天，流失31-90天，深度流失>90天）更新 `activity_tag`。

对于本次判定中**新进入"待唤醒"**状态的用户（即上一次是"活跃"，这一次变成"待唤醒"），自动创建一条 `broadcast_tasks` 记录：模板类型为"唤醒"，`target_filter` 为本批次新进入待唤醒的用户ID列表，`scheduled_time` 为立即或按运营配置的延迟时间。

## 4. 积分月度批次过期处理（每日一次，或至少每月1号执行一次，建议每日跑以确保及时性）

扫描 `points_monthly_batches` 表中 `status=有效 且 expire_at <= 当前时间` 的记录：

- 计算剩余未消费积分：`remaining = points_earned_total - points_consumed_total`
- 若 `remaining > 0`：写入一条 `points_ledger` 记录（`change_type=过期清零`，`amount = -remaining`），更新对应 `users.points_balance -= remaining`
- 将该批次 `status` 置为"已过期"，`points_consumed_total` 补齐为等于 `points_earned_total`

**到期提醒（同一任务内或单独任务，建议提前7天）**：扫描 `expire_at` 在未来7天内、且 `remaining > 0` 的批次，去重用户后触发"积分到期提醒"消息模板推送（可直接发送私聊消息，或走 broadcast_tasks 队列）。

## 5. 积分消费的FIFO扣减逻辑（被兑换/后台调整为负时触发，非独立定时任务，但与上面月度过期逻辑共用同一套批次表，故一并说明）

当用户积分被消费（提现兑换核销、后台调整为负）时：

1. 按 `batch_month` 升序取出该用户所有 `status=有效` 的批次
2. 依次从每个批次的剩余额度（`points_earned_total - points_consumed_total`）中扣减，直到扣满所需消费数量
3. 更新涉及批次的 `points_consumed_total`
4. 若消费数量超过所有有效批次剩余总和（理论上不应发生，因为 `users.points_balance` 应与批次剩余总和一致），需报错并触发对账告警，不允许透支

## 6. 月度邀请排行榜结算与推送（每月1号凌晨执行，结算上一自然月）

1. 统计上一自然月内，按 `invited_by_l1` 分组统计各用户的新增注册数（即当月直接邀请了多少新会员）
2. 按数量降序排名，取前 `points_config.leaderboard_top_n` 名
3. 写入 `leaderboard_snapshot`（period=上月，user_id，rank，invite_count_this_period，reward_points）
4. 给上榜用户按名次发放对应奖励，写入 `points_ledger`（change_type=排行榜奖励），同步 `points_monthly_batches`
5. 创建 `broadcast_tasks`：模板类型"月度排行榜"，`target_filter=全体会员`（已确认推送范围为全体，不是只通知上榜者），内容展示本月榜单前N名

## 7. 群发任务队列消费（常驻任务/worker，非固定周期）

持续消费 `broadcast_tasks` 中 `status=待发送` 且 `scheduled_time<=当前时间` 的任务：按 `target_filter` 解析目标用户列表，逐条发送消息（控制在 Telegram 限速内，约每秒30条到不同用户），更新 `sent_count`，记录点击（如消息内按钮带追踪参数）更新 `click_count`，全部发完后 `status=已完成`，异常情况下标记`失败`并记录错误详情供人工排查。
