## 项目概述

Telegram 会员裂变机器人：用户关注机器人自动注册为会员，通过签到、邀请获得积分，积分仅可联系客服兑换（无法打通网站数据，走人工核销流程）。核心目标是提升网站日活、通过3级邀请返佣驱动会员裂变增长。后台是一套 Laravel + Filament 管理系统，覆盖数据面板、会员管理、消息模板、积分配置、账变记录、系统管理、提现申请管理、机器人配置、域名配置九大模块。

## 技术栈（已确定，不要更换）

PHP + Laravel + MySQL；机器人库用 Nutgram（`nutgram/laravel`）；后台管理用 Filament；队列用 Laravel Queue（推荐 Redis 驱动，简化部署也可用数据库驱动）；定时任务用 Laravel Task Scheduling；海报生成用 Intervention Image + `endroid/qr-code`；机器人用长轮询（不用 webhook）；机器人轮询进程与队列 worker 用 Supervisor 常驻守护，不要挂在常规 PHP-FPM 请求池里。

## 不可违反的设计约束

以下是头脑风暴阶段反复讨论确认的关键决策，实现时必须遵守，不要"顺手优化"改掉：

- **邀请返佣发放时机**：仅在被邀请人完成首次签到后才发放，不做"关注即发放"。这是刻意的防刷设计，不是遗漏。
- **3级返佣关系存储**：用 `users.invited_by_l1/l2/l3` 三个冗余字段，注册时一次性写入，不要改成递归查询或路径字符串方案。
- **积分过期**：用 `points_monthly_batches` 按月度批次简化管理（月度粒度FIFO），不要做逐笔精确 FIFO，这是已经讨论过并有意简化的方案。
- **枚举字段**：01文档里标注的 `enum(...)` 不要用 MySQL 原生 ENUM 类型，用 varchar + PHP 8.1 原生 enum class 做应用层约束，避免后续加枚举值要改表结构。
- **邀请短链**：自建短链服务，不接第三方短链平台；生成短链时从"启用"状态的域名池中随机/轮询分配域名，不要所有用户共用同一个域名。
- **机器人容灾**：`bots` 表任何时刻只能有一条 `is_active=true`；切换当前生效机器人后，历史短链应自动指向新机器人，不需要重新生成短链或海报。
- **月度排行榜**：排名维度是"本月新增邀请数"，不是历史累计总数；结算后推送给全体会员，不是只通知上榜者。
- **提现兑换**：无法打通网站数据，走"用户提交兑换申请记录 → 客服人工核实 → 管理员后台标记完成"的流程，不要假设可以自动化发放。
- **群发限速**：任何批量发送消息的地方都必须走限速队列（约每秒30条到不同用户），不允许无限制并发调用 Telegram API。
- **Bot Token**：加密存储（如 Laravel 的 `encrypted` cast），不要明文入库或写进 `.env` 以外的地方。
- **安装向导**：是否已安装的判断用文件锁 `storage/app/installed.lock`，不要改成查数据库判断（环境检测/数据库配置阶段数据库可能还没配好）。安装完成后 `/install` 路由组必须失效（重定向走），不能保留可重复触发的入口，这是刻意的安全设计，不是待办。

## 开发规范

- 代码风格遵循 PSR-12，提交前跑 `./vendor/bin/pint`
- 遵循 Laravel 默认约定（表名复数、迁移文件命名、Eloquent 关系命名等），不要引入非常规的自定义框架层
- 业务逻辑（如返佣计算、积分批次扣减、活跃度层级判定）建议放在 `app/Services/` 下的 Service 类里，不要直接堆在 Controller 或 Nutgram 的 handler 里，方便单独写单元测试
- 定时任务、队列任务对应04文档的每一项，建议一个任务对应一个独立的 `app/Console/Commands` 或 `app/Jobs` 类，命名和04文档的任务名对应，方便交叉核对

## Git 工作流（已确认：每完成一个小节自动提交并推送）

- 每完成一个可独立验证的功能小节（例如：02文档里的"签到"整个交互逻辑、04文档里的"机器人健康检测"这一个定时任务、07文档里安装向导的某一步骤页面），先本地跑一遍 `php artisan test` 和 `./vendor/bin/pint`，确认通过后再提交，不要把跑不过测试或有明显语法问题的代码推上去。
- 提交粒度按"一个小节/一个子功能"为单位，不要把多个不相关的功能改动堆在同一个 commit 里，也不要反过来把一个小功能拆成很多个无意义的碎 commit。
- Commit message 用 Conventional Commits 风格（`feat: `、`fix: `、`chore: `、`docs: ` 等前缀），并在描述里点出对应规格文档的章节，例如：`feat: 实现签到功能与首次签到返佣触发 (对应02文档"签到"一节)`，方便回溯某个功能对应文档的哪一部分。
- 提交后立即 `git push` 到当前分支（默认直接在 `main` 上开发，除非另有分支策略说明）。推送前如果发现远程有领先的新提交，用 `git pull --rebase` 同步后再推，**不要用 `git push --force`**，避免覆盖掉可能存在的其他改动。
- 前提：仓库需要已经关联好远程地址（GitHub/Gitee等），且本机已有推送权限。这是开发环境准备阶段需要确认好的前置条件，不是 Claude Code 能自己解决的部分。

## 常用命令

```
php artisan serve                # 本地启动（后台管理界面）
php artisan migrate              # 执行迁移
php artisan queue:work           # 启动队列worker（本地测试用，生产环境交给Supervisor）
php artisan schedule:work        # 本地模拟定时任务触发（生产环境走系统cron）
php artisan test                 # 跑测试
./vendor/bin/pint                # 代码风格检查/修复
```
