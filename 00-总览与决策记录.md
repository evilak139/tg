## 项目背景

会员关注 Telegram 机器人后自动注册为会员，通过"签到 + 邀请"积累积分，积分可联系客服兑换。核心目标：提升网站日活、通过邀请裂变机制驱动会员增长。后台支持对会员分层运营（活跃度/身份等级），支持图文消息模板化群发，并针对 Telegram 机器人容易被封禁的风险做了多机器人、多域名的容灾设计。

## 技术栈（已确认：PHP + Laravel + MySQL）

| 层 | 选型 | 说明 |
|---|---|---|
| 框架 | Laravel | Eloquent ORM、内置队列、内置任务调度、迁移文件管理，01文档表结构直接对应 migration |
| 机器人库 | Nutgram（`nutgram/laravel` 官方Laravel集成包） | 对多步骤交互（邀请/签到/提现的状态流转）支持较好，中间件机制完善；若团队更熟悉 `telegram-bot-sdk` 亦可替换，只是状态管理需自行多做一层 |
| 数据库 | MySQL + Eloquent | |
| 队列 | Laravel Queue，建议用 Redis 驱动（性能更好），预算/复杂度敏感可用数据库驱动（免加 Redis，建表即可） | 群发限速用 Laravel Queue 自带的 `Illuminate\Queue\Middleware\RateLimited` 中间件卡住发送速率，对应06文档"Telegram限速"要求 |
| 定时任务 | Laravel Task Scheduling | 服务器只需一条 cron：`* * * * * php artisan schedule:run`，04文档各任务的具体频率在代码里配置 |
| 后台管理面板 | Filament | 声明式生成CRUD界面，"会员列表""账变记录""提现申请管理""机器人配置""域名配置""消息模板管理"这类模块用 Filament Resource 搭建；"数据面板"统计卡片/图表用 Filament Widget |
| 海报生成 | Intervention Image + `endroid/qr-code` | 邀请海报合成头像/积分/邀请人数/二维码 |
| 机器人收发模式 | 长轮询（long polling），非 webhook | 便于"一键切换当前生效机器人"这个容灾需求（切换只是换token继续轮询，不用重新注册webhook） |
| 进程管理 | Supervisor | PHP 默认是短生命周期请求模型，机器人轮询逻辑需要写成独立 artisan 命令，用 Supervisor 常驻守护（挂掉自动重启），与常规处理网页请求的 PHP-FPM 池子是两种运行模式，部署时需分开配置 |
| 部署 | Docker Compose | Nginx + PHP-FPM（后台管理web界面）+ MySQL + Redis（可选）+ Supervisor（管机器人轮询进程、队列worker进程） |
| 安全 | Bot Token 需加密存储（如用 Laravel 的 `encrypted` cast），不要明文入库 | |

**实现细节提醒**：01文档中标注的 `enum(...)` 字段类型，实现时不建议用 MySQL 原生 ENUM 类型（后续增删枚举值需要改表结构，不够灵活），建议用普通字符串字段 + Laravel 的 PHP Enum class（PHP 8.1+ 原生枚举）或应用层常量校验来实现，migration 里就是 varchar，约束交给应用层。

## 已确认的核心参数汇总（供快速查阅，详细逻辑见对应分文档）

| 项目 | 确认结果 |
|---|---|
| 邀请返佣层级 | 3级返佣，默认比例：一级10分 / 二级3分 / 三级1分（后台可配置） |
| 邀请返佣发放时机 | 被邀请人完成首次签到后才发放，不做"关注即发放" |
| 邀请里程碑奖励 | 累计邀请满5/20/100人解锁额外奖励，金额后台配置 |
| 邀请排行榜 | 月榜，按本月新增邀请数排名（非历史累计），每月重置 |
| 排行榜推送范围 | 推送给全体会员（不是只通知上榜者），制造围观效应 |
| 积分消耗渠道 | 仅"兑换"一种，不做积分商城/多元消耗 |
| 提现兑换流程 | 无法打通网站数据，走"用户提交兑换申请记录 + 客服人工核实 + 管理员后台标记完成"流程 |
| 积分有效期 | 开启，按"月度批次"简化管理（非逐笔精确FIFO），默认12个月未使用清零 |
| 任务体系（每周任务/打卡等） | 不做，平台功能暂不支持 |
| 会员层级 | 拆成两条独立的线：活跃度层级（活跃/待唤醒/流失/深度流失，运营用）+ 身份等级（注册会员/邀请达人/团队长/VIP大使，用户可见） |
| 邀请链接形式 | 后台自建短链（不用第三方短链服务），格式为 `{域名}/r/{短码}`，302跳转到当前生效机器人 |
| 机器人容灾 | 支持多机器人配置，健康检测+一键切换"当前生效机器人"，切换后历史短链自动生效，无需重新生成 |
| 域名容灾 | 支持域名池，短链生成时从"启用"域名中随机/轮询分配，非全部使用同一域名；域名被封的历史短链无法后台恢复，因此靠源头分散风险 |

## 待开发过程中留意的坑（来自头脑风暴过程中的讨论）

- 已兑换 vs 未兑换积分这个数据面板指标，数据来源是"提现申请管理"模块的记录状态，必须先有"生成申请记录"这个动作，否则这个指标无从统计。
- 排行榜排名维度务必用"本月新增邀请数"而非"历史累计总数"，否则新用户永远无法进入前列，会打击后来者积极性。
- 域名被封和机器人被封的容灾逻辑不一样：机器人可以后台切换跳转目标做到历史链接无损切换；域名被封是域名层面失效，历史用该域名的短链无法恢复，只能靠"生成时随机分配到多个域名"来分散风险，不能靠切换补救。
