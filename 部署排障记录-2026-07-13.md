# 2026-07-13 部署排障记录

服务器：宝塔面板，Linux，站点目录 `/www/wwwroot/s.trxs.io`，域名 `s.trxs.io`（Cloudflare 代理）。
本文档记录今天从"代码部署上线"到"广播消息/邀请链接能正常工作"过程中遇到的全部问题、根因和解决方式，方便以后重新部署或者出现类似问题时查阅。

---

## 一、代码部署基础

### 1.1 首次部署缺 vendor 目录

**现象**：打开网站报错 `Failed to open stream: vendor/autoload.php`。

**原因**：`vendor/` 目录被 `.gitignore` 排除，git 同步代码不会带上依赖包。

**解决**：
```bash
cd /www/wwwroot/s.trxs.io
composer install --no-dev --optimize-autoloader
```
如果国内网络慢，换阿里云镜像：
```bash
composer config -g repo.packagist composer https://mirrors.aliyun.com/composer/
```

### 1.2 每次 `git pull` 之后必做的收尾动作

这是最容易漏掉、也是今天踩坑最多的地方——**PHP 常驻进程（队列 worker、机器人长轮询）不会自动感知代码更新**，git pull 完了不重启这两个进程，它们会一直执行内存里的旧代码（今天"自定义"模板类型报错就是这个原因，见下文 4.1）。

标准部署流程：
```bash
cd /www/wwwroot/s.trxs.io
git pull
composer install --no-dev --optimize-autoloader   # 只有依赖变了才需要
php artisan migrate --force                         # 有新迁移才需要
php artisan optimize:clear                          # 清所有缓存（config/route/view）

# 重启两个常驻进程（重点！）
kill $(ps aux | grep "queue:work" | grep -v grep | awk '{print $2}')
kill $(ps aux | grep "nutgram:run" | grep -v grep | awk '{print $2}')
nohup php artisan queue:work --tries=3 --sleep=3 >> storage/logs/queue-worker.log 2>&1 &
nohup php artisan nutgram:run >> storage/logs/bot-polling.log 2>&1 &
```

> 后面配好 Supervisor（见第五节）之后，重启命令会变成 `supervisorctl restart tg-queue-worker tg-bot-polling`，更省心也更安全（进程意外退出会自动拉起）。

## 二、后台白屏 / 静态资源问题

### 2.1 后台完全白屏，只有标题

**现象**：`/admin` 打开只显示标题文字，没有任何界面元素。

**排查**：浏览器控制台看到 Livewire 的 JS 资源 `/livewire-xxxx/livewire.js` 返回 404，但浏览器直接访问这个路径又"存在"（说明是网关/Nginx 层的问题，不是 Laravel 路由本身的问题）。

**根因**：Nginx 里处理 `.js`/`.css` 静态资源的 location 块没有 `try_files` 兜底，Livewire 这个路径长得像静态文件但其实是动态路由，Nginx 直接给它 404 掉了，请求根本没进到 PHP-FPM。

**解决**：给站点 Nginx 配置里 `.js`/`.css` 的 location 块加 `try_files` 回退到 `index.php`：
```nginx
location ~ .*\.(js|css)?$
{
    try_files $uri $uri/ /index.php?$query_string;
    expires 12h;
    error_log /dev/null;
    access_log /dev/null;
}
```
改完 `nginx -s reload` 或者宝塔面板里重载配置。

### 2.2 改完 Nginx 还是 404 —— Cloudflare 缓存了旧的 404

**现象**：Nginx 配置改完、`optimize:clear` 也跑了，浏览器还是 404。

**根因**：`curl -I` 看响应头发现 `server: cloudflare`——Cloudflare 默认会按扩展名缓存 `.js`/`.css` 响应（包括错误响应），Nginx 层已经修好了，但 CDN 边缘节点还在吐旧的缓存内容。

**解决**：Cloudflare 后台 → Caching → Configuration → **Purge Everything**，清完立刻生效。

## 三、机器人 / SSL 相关

### 3.1（本地开发环境）PHP 请求 Telegram API 报 SSL 证书错误

**现象**：`SSL certificate problem: self-signed certificate in certificate chain`，但系统自带的 `curl.exe` 和浏览器访问 Telegram API 都正常。

**根因**：Windows/宝塔环境下 PHP 的 `curl.cainfo` / `openssl.cafile` 没配置，PHP 发起的 HTTPS 请求（Guzzle/Nutgram 底层用的）拿不到证书链去验证，而系统级 curl/浏览器是用操作系统自带的证书库，不受这个影响。

**解决**（该服务器专属，Linux 生产服务器一般不会有这个问题，Linux 系统证书链通常是配好的）：
1. 下载证书包到 `php.ini` 所在目录：`C:\BtSoft\php\83\ssl\cacert.pem`（从 https://curl.se/ca/cacert.pem 下载）
2. `php.ini` 里加两行：
   ```ini
   curl.cainfo = "C:\BtSoft\php\83\ssl\cacert.pem"
   openssl.cafile = "C:\BtSoft\php\83\ssl\cacert.pem"
   ```
3. 重启 PHP-FPM

### 3.2 一键部署之后机器人前端菜单没变化

**现象**：点了后台"一键部署"，Telegram 里 `/start` 出来的还是老样子，没有邀请/签到/提现/我的按钮。

**根因**："一键部署"只是调用 Telegram 的 `setMyCommands` 接口，推送命令菜单说明，**不会启动机器人本身的长轮询进程**。机器人要真正响应消息、显示菜单，必须有 `php artisan nutgram:run` 这个进程常驻运行。

**解决**：见第五节，把 `nutgram:run` 用 Supervisor（或至少手动 `nohup`）跑起来。

## 四、消息模板 / 图片

### 4.1 消息模板编辑页 TypeError

**现象**：点开某条已有消息模板的编辑页报错：
```
variableHelperText(): Argument #1 ($type) must be of type ?string, App\Enums\MessageTemplateType given
```

**根因**：Eloquent 对 `type` 字段做了枚举 cast——**新建/表单交互**时 Filament 传过来的是原始字符串，但**编辑已有记录**时从数据库读出来的是枚举实例，两种类型都要接住。原来的代码只处理了字符串这一种情况。

**解决**：加了一个统一的类型归一化方法（`resolveType()`），两种输入都转成枚举实例再处理。已提交（`fdbf37b`），并补了一条专门打开"每一种已存在模板的编辑页"的回归测试，防止同类 bug 再次漏测（原来的测试只覆盖了列表页，没覆盖编辑页）。

### 4.2 消息模板新增自定义类型

**需求**：后台原来只能编辑 05 文档里固定的 7 种触发型模板（欢迎/邀请/签到/我的/唤醒/积分到期提醒/月度排行榜），不能新建。用户希望能新增模板（主要给"群发消息"功能选文案用）。

**方案**：加了一个不挂任何自动触发场景的类型 `自定义`（Custom）。后台可以新建/删除这个类型的模板，原来 7 种系统类型依旧只能编辑、不能新建或删除（避免同一触发场景出现多条模板，渲染时不知道用哪条）。已提交（`89d48ad`）。

### 4.3 模板配图上传后链接 404

**现象**：上传配图后生成的链接 `https://s.trxs.io/storage/message-templates/xxx.png` 打开是 404，实际文件却在服务器的 `storage/app/private/message-templates/` 目录下。

**根因（两层）**：
1. **磁盘选错了**：Laravel 11 起，默认的 `local` 磁盘根目录改成了 `storage/app/private`（不对外公开），Filament 的图片上传组件没有显式指定磁盘时，会跟着 `FILESYSTEM_DISK`（默认值 `local`）走到这个私有磁盘上，存进去的文件天生就没法通过 URL 访问到。
2. **就算磁盘对了，字段存的也只是相对路径**：`message_templates.image_url` 存的是类似 `message-templates/xxx.png` 这样的相对路径，不是完整 URL，直接拿去给 Telegram 的 `sendPhoto` 用是不可用的（Telegram 需要能直接抓取的绝对 URL），必须转换成完整 URL 再用。

**解决**（已提交 `39a6de6`）：
- 图片上传组件、表格缩略图组件都显式指定 `->disk('public')`，让文件落到 `storage/app/public/` 下；
- `MessageTemplateRenderer` 渲染时用 `Storage::disk('public')->url($path)` 把相对路径转成完整 URL 再返回。

**配套的服务器操作**（这部分代码解决不了，必须手动做）：
```bash
cd /www/wwwroot/s.trxs.io

# 1. 建 public/storage 软链（Nginx 静态资源服务需要这个软链才能读到storage/app/public里的文件）
php artisan storage:link

# 2. 之前存错位置（private目录）的旧文件可以删掉，重新上传即可
rm -f storage/app/private/message-templates/*.png
```

**为什么不直接把文件存到 `public/` 目录下**：`public/` 按 Laravel 约定只放"部署时确定的东西"（入口文件、编译后的前端资源），用户上传的内容不应该混进去，避免以后代码更新流程误删/覆盖用户数据。`storage/app/public` + `public/storage` 软链这套组合，能做到"物理存放在 storage 里、但 Nginx 能像访问 public 目录一样访问到"，两全其美。

## 五、常驻进程 / 定时任务（重点，必须长期维护）

系统里有三类东西**必须常驻运行**，缺一个都会导致"消息发不出去"这种问题：

| 进程 | 作用 | 命令 |
|---|---|---|
| 队列 worker | 处理群发消息、返佣发放等所有 `dispatch()` 出去的异步任务 | `php artisan queue:work --tries=3 --sleep=3` |
| 机器人长轮询 | 让机器人真正响应 Telegram 消息（回复 /start、签到、菜单按钮等） | `php artisan nutgram:run` |
| 定时任务调度 | 每分钟检查一次有没有到点该跑的定时任务（比如"到点该发的群发任务"） | 需要系统 cron 每分钟触发一次 `php artisan schedule:run` |

### 5.1 检查 cron 有没有配置调度器

```bash
crontab -l
```
如果没有一条是 `php artisan schedule:run`，去宝塔面板「计划任务」新建一条：
- 任务类型：Shell 脚本
- 执行周期：每 1 分钟
- 脚本内容：
```bash
cd /www/wwwroot/s.trxs.io && php artisan schedule:run >> /dev/null 2>&1
```

### 5.2 队列驱动：从 Redis 改成了数据库驱动

**现象**：跑 `php artisan app:dispatch-due-broadcast-tasks` 报错 `Class "Redis" not found`，导致所有队列任务（不只是广播消息，邀请返佣、积分批次等所有异步任务）从投递那一步就直接失败，广播任务永远卡在"待发送"、目标人数 0。

**根因**：`.env` 里 `QUEUE_CONNECTION=redis`，但服务器 PHP 没装 `phpredis` 扩展。

**解决**（已提交 `afda3d7`，`.env.example` 默认值已改）：`.env` 里改成：
```
QUEUE_CONNECTION=database
```
`jobs`/`failed_jobs` 表是 Laravel 默认迁移自带的，不需要额外建表，`migrate` 就有。

> 如果以后想换回 Redis 队列（并发量大了 database 驱动性能会成为瓶颈），需要在宝塔"软件商店"装 `phpredis` 这个 PHP 扩展并重启 PHP-FPM，再把 `QUEUE_CONNECTION` 改回 `redis`。

### 5.3 时区不一致

**现象**：怀疑服务器时间和 PHP 时间不一致（实际排查下来，系统时间和 PHP CLI 时间本身是一致的，都是 `Asia/Shanghai`/CST）。

**真正的问题**：`config/app.php` 里 `'timezone'` 硬编码成了 `'UTC'`，跟 `date_default_timezone_get()`（PHP 的时区设置）是两回事——**Laravel 内部 `now()`/Carbon 相关的时间计算走的是 `config('app.timezone')`，不会自动跟随服务器/PHP 的时区设置**。

**解决**（已提交 `ab233a2`）：
```php
// config/app.php
'timezone' => env('APP_TIMEZONE', 'Asia/Shanghai'),
```
`.env` 里加一行：
```
APP_TIMEZONE=Asia/Shanghai
```

### 5.4 用宝塔"进程守护管理器"把两个进程真正常驻起来（推荐，还没做）

目前 `queue:work` 和 `nutgram:run` 都是用 `nohup ... &` 手动起的临时方案，服务器重启、进程意外退出都不会自动拉起来。正式的做法：

1. 宝塔「软件商店」搜索安装「进程守护管理器」
2. 添加守护进程 1（队列 worker）：
   - 名称：`tg-queue-worker`
   - 运行目录：`/www/wwwroot/s.trxs.io`
   - 启动命令：`php artisan queue:work --tries=3 --sleep=3`
3. 添加守护进程 2（机器人长轮询）：
   - 名称：`tg-bot-polling`
   - 运行目录：`/www/wwwroot/s.trxs.io`
   - 启动命令：`php artisan nutgram:run`

配好之后，每次部署新代码，重启命令变成：
```bash
supervisorctl restart tg-queue-worker tg-bot-polling
```

## 六、广播消息（群发）功能相关

### 6.1 "立即发送"始终被存成"定时发送"

**现象**：不管在群发消息表单里选"立即发送"还是"定时发送"，列表页看到的都是"定时发送"。

**根因**：列表页判断是"立即"还是"定时"的逻辑用了 Carbon 的 `diffInSeconds()`，这个方法在项目所用的 Carbon 版本里是**带符号的**——顺序反了会一直算出负数，导致 `abs()` 判断前的逻辑永远走不到"立即发送"分支。

**解决**（已提交 `4a0c2a1`）：加 `abs()`。同时补上了漏掉的"定时发送任务删除按钮"（`DeleteAction` 原来没有接到 Resource 的 `canDelete()` 上，Filament 不会自动帮你接）。

### 6.2 广播任务一直卡在"待发送"，目标人数 0

**现象**：群发任务创建成功、显示"立即发送"，但状态永远是"待发送"，目标人数/已发送都是 0。

**根因**：见 5.2（队列跑不起来）+ 5.1（cron 没配）。这两个基础设施没配好，`DispatchDueBroadcastTasks` 定时任务和后续的 `SendBroadcastMessageJob` 队列任务压根没被执行过。

### 6.3 状态显示"已完成"，但"已发送"是 0

**现象**：任务目标人数显示正常（比如 3 人），状态也变成了"已完成"，但"已发送"一直是 0，日志里能看到具体报错。

**遇到过的两种具体报错**：

1. **`Bad Request: wrong HTTP URL specified`**——群发的模板带图片，生成的图片链接用了错的 `APP_URL`（比如还是 `http://localhost`），Telegram 服务器抓不到这个链接。
   ```bash
   # 检查
   grep APP_URL /www/wwwroot/s.trxs.io/.env
   # 改成真实域名
   sed -i 's|^APP_URL=.*|APP_URL=https://s.trxs.io|' /www/wwwroot/s.trxs.io/.env
   php artisan config:clear
   ```

2. **`"自定义" is not a valid backing value for enum ...`**——见 1.2，`queue:work` 进程还在跑加"自定义"模板类型之前的旧代码，重启进程即可。

> 单条发送失败会被 `try/catch` 悄悄吞掉、只记 warning 日志，不会让整个批次任务失败（这是故意的，避免一个用户发送失败拖累整批），**所以后台看不出具体原因，必须去日志里查**：
> ```bash
> grep "群发任务" storage/logs/laravel.log | tail -20
> ```

## 七、邀请链接 / 短链

### 7.1 短链跳转路由从未实现过（已修复）

**现象**：生成的邀请短链（`{域名}/r/{短码}` 格式）打开是 404。

**根因**：短链的"生成"逻辑（`InviteLinkService`）一直是完整实现的，但对应的"跳转"路由 `/r/{短码}` 代码里从来没写过（当初留了个 `// TODO` 性质的注释说"后续实现"，后续一直没跟上）。

**解决**（已提交 `c923e1b`）：新增了 `ShortLinkController` + `/r/{shortCode}` 路由。逻辑：按短码查 `invite_links` 表，实时查当前 `is_active=true` 的机器人（不是把机器人信息写死进短链记录），302 跳转到 `https://t.me/{机器人用户名}?start={邀请人的user_id}`，命中一次 `click_count` 计数 +1。这样"切换当前生效机器人"之后，历史短链会自动跳到新机器人，不需要重新生成短链或海报（对应产品早期决策）。

### 7.2 邀请链接暂时改用 Telegram 原生深链（产品决定）

**背景**：自建短链系统依赖"后台配置至少一个启用状态的域名"，域名配置/DNS 还没搞定之前，邀请功能完全不可用。产品决定先绕过域名配置，直接用 Telegram 原生链接顶一阵子。

**改动**（已提交 `386ce93`）：
- `{邀请链接}` 变量、邀请海报生成的链接，现在固定是 `https://t.me/{当前生效机器人用户名}?start={邀请人ID}`，不再要求先配置域名。
- 自建短链系统的代码（`/r/{短码}` 跳转路由、域名池分配逻辑）**都还在，没删**，`invite_links` 表的 `domain_id` 字段改成了可为空——域名配置齐了之后，短链记录照常正常生成（为将来切回去保留数据），只是暂时不用来生成展示给用户的链接。以后想切回自建短链系统，只需要改 `InviteLinkService::buildUrl()` 这一处代码。
- **已知副作用**：走 Telegram 原生链接后，用户点击不经过我们的服务器，邀请链接的"点击数"统计这个功能暂时会一直是 0（切回自建短链系统后会恢复）。

## 八、以后再遇到"功能不生效"类问题的排查顺序建议

1. **先看有没有报错日志**：`tail -f storage/logs/laravel.log`，或者对应功能的关键字 `grep`。
2. **确认三大常驻服务都在跑**：
   ```bash
   ps aux | grep -E "queue:work|nutgram:run" | grep -v grep
   crontab -l
   ```
3. **确认代码是最新的、常驻进程也重启过**：`git log -1` 看当前 commit，对比常驻进程的启动时间（`ps -eo pid,lstart,cmd | grep queue:work`）是不是在这次 `git pull` 之后。
4. **确认配置缓存是干净的**：`php artisan optimize:clear`。

## 九、2026-07-15 换机器人后"菜单消失"排查全过程（新服务器 `tg` 站点）

这次排查花了很长时间，暴露了好几个叠在一起的问题，从上到下按实际排查顺序记录，代码层面能补的坑已经补上（见本节最后"已落地的机制改进"）。

### 9.1 环境准备阶段的坑（跟老服务器踩过的同类，重新踩了一遍）

- `disable_functions` 把 `pcntl_signal`/`pcntl_async_signals`/`pcntl_alarm`/`symlink`/`proc_open` 等一起禁了，导致 `queue:work`、`nutgram:run`、`storage:link`、`schedule:run` 全部不能正常跑。解决方式同老服务器：编辑 `php --ini` 指向的配置文件，把这几个函数从 `disable_functions` 里删掉，重启 PHP。
- 命令行默认 `php` 指向的是系统自带的 8.0，不是宝塔装的 8.3，导致 `composer install` 生成的 `vendor` 用不了。解决：`ln -sf /www/server/php/83/bin/php /usr/bin/php`，crontab/常驻进程启动命令也都要用这个绝对路径，不能依赖裸 `php`。
- cron 没配 `schedule:run`，群发任务一直卡"待发送"。

### 9.2 切换"当前生效机器人"之后，菜单彻底消失

**根因**：`AppServiceProvider::useActiveBotToken()` 只在进程**启动那一刻**读一次数据库里的 `is_active` 机器人 token，写进 `config('nutgram.token')`。`nutgram:run` 是长驻进程，容器只在进程启动时初始化一次 `Nutgram` 单例，运行期间**永远不会重新读取**这个 token——切换"当前生效机器人"只改了数据库字段，正在跑的轮询进程完全不知情，还在用切换前的旧 token 连 Telegram。

**当时怎么处理的**：手动 kill 掉进程重新 `nohup` 启动。

**踩到的次生坑**：这台服务器上 `nutgram:run` 其实是用宝塔"进程守护管理器"托管的（`tg-bot-polling`），手动 `kill` 只会被守护进程立刻拉起一个新的，跟自己手动 `nohup` 起的那个互相抢同一个 token，直接 `409 Conflict: terminated by other getUpdates request`。**以后重启这个进程只能用宝塔面板"进程守护管理器"里的"重启"按钮，不要手动 kill。**

### 9.3 进程不冲突之后，`/start` 依然完全没反应，两个日志文件都是空的

这是最难查的一段。关键发现：

- `vendor/nutgram/nutgram/src/RunningMode/Polling.php` 的 `fire()` 方法里，handler 抛出的异常是 `fwrite(STDERR, "$e\n")`，**不会进 Laravel 的任何日志通道**，`config('nutgram.log_channel')` 也管不到这条路径。这也是为什么排查过程中不管怎么调日志通道、`grep laravel.log`，都看不到任何异常痕迹——异常确实发生了，只是被写到了一个我们一直没看对地方的地方。
- 临时在 `StartHandler` 里加了逐步打点日志（用 `Log::channel('single')` 强制绕开 `nutgram.log_channel` 配置），才最终揪出真正的异常：

  ```
  SQLSTATE[23000]: Integrity constraint violation: 1048 Column 'domain_id' cannot be null
  insert into `invite_links` (`user_id`, `domain_id`, `short_code`, `created_at`) values (...)
  ```

  仓库里其实早就有一条迁移 `2026_07_13_070000_make_invite_links_domain_id_nullable.php` 把这一列改成了可为空（对应"邀请链接暂时改用 Telegram 原生深链接"那个产品决定），但**这台服务器的数据库从来没跑过这条迁移**，字段实际还是旧的 NOT NULL 结构。渲染欢迎消息时要生成邀请链接变量，一建 `invite_links` 记录就因为约束冲突而抛异常，异常又被 Nutgram 静默吞进 STDERR，最终表现就是"机器人完全没反应，两个日志文件都是空的"。

  **解决**：`php artisan migrate --force`。

### 9.4 已落地的机制改进（代码层面，防止同类问题再发生）

1. **全局异常兜底**（`routes/telegram.php`）：注册了 `$bot->onException(...)`，以后任何一个 handler 内部抛异常，都会显式落进 `storage/logs/laravel.log`（用 `Log::channel('single')`，不依赖 `nutgram.log_channel` 配置），不会再出现"进程活着、日志全空、用户端毫无反应"这种排查了大半天都摸不到头绪的情况。同时给用户回一句"服务暂时出现问题，请稍后再试"，不再是死一般的沉默。见 `tests/Feature/Telegram/GlobalExceptionHandlerTest.php`。
2. **后台首页自动检测"有没有迁移没跑"**（`SetupReminder` 组件）：以后 `git pull` 完忘了跑 `migrate`，一进后台首页就会有醒目的红色提示"检测到 N 个数据库迁移尚未执行"，不用等到某个具体功能（比如这次的邀请链接）踩坑才发现。
3. **"设为当前生效"确认弹窗和成功提示都补充了提醒**：明确告诉管理员切换后必须去服务器重启机器人长轮询进程（并且要用进程守护管理器的"重启"按钮，不要手动 kill），否则新机器人不会有任何响应。
