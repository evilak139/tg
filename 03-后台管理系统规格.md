# 03 后台管理系统规格

> 对应数据表见 01-数据库设计.md

## 3.1 数据面板

指标及计算口径：

- 总会员数 / 昨日新增 / 今日新增：按 `users.register_time` 统计
- 日活/周活/月活：按 `users.last_active_time` 落在对应时间窗口内的去重用户数（签到、邀请等行为均更新 last_active_time）
- 次日留存率 / 7日留存率：按注册次日/第7日是否有活跃行为计算的标准留存口径
- K因子：`本周期新增注册数中，由老用户邀请产生的数量 / 本周期活跃老用户数`，用于衡量裂变效果
- 总发放/昨日发放/今日发放（积分）：按 points_ledger 中 amount>0 的记录求和
- 已兑换积分 vs 未兑换积分余额：已兑换=withdraw_requests.status=已完成 的 points_amount 求和；未兑换余额=所有用户 points_balance 求和减去已过期清零部分（或直接展示"当前全体用户 points_balance 总和"作为系统积分负债）
- 群发消息送达率、点击率：来自 broadcast_tasks 的 sent_count / total_target_count，click_count / sent_count
- 当前生效机器人状态：读取 bots 表 is_active=true 的记录及其 status
- 域名池健康状态：domains 表按 status 分组计数
- 短链访问量统计（按域名拆分）：来自 invite_links.click_count 按 domain_id 聚合

## 3.2 会员列表

筛选项：

- 按邀请人筛选（输入某用户ID，查看其 `invited_by_l1/l2/l3` 为该用户的完整邀请树）
- 按注册时间区间筛选
- 按活跃度层级筛选（活跃/待唤醒/流失/深度流失）
- 按身份等级筛选

操作：

- 手动调整积分（写入 points_ledger，change_type=后台调整，需记录 operator）
- 拉黑/风控标记（更新 users.status）

## 3.3 消息模板管理

CRUD 界面，管理 message_templates 表的7种类型：欢迎、邀请、签到、我的、唤醒、积分到期提醒、月度排行榜。每种类型支持富文本+图片+变量占位符编辑，变量清单见05文档。保存时校验变量拼写是否在允许清单内，避免运营手滑打错变量导致线上展示原始占位符文本。

## 3.4 积分配置

对应 `points_config` 表的可视化编辑界面，涵盖：签到基础积分及递增规则、邀请1/2/3级积分、里程碑奖励金额、积分有效期月数（当前为12，按月度批次简化方案）、兑换比例、最低提现门槛、新账号提现限制天数、活跃度层级阈值、排行榜奖励覆盖前N名。

## 3.5 账变记录

展示 points_ledger 全量流水，支持按用户/类型/时间筛选，支持导出为 CSV/Excel。

## 3.6 系统管理

- 管理员权限分级：基于 admin_users.role（超级管理员/客服/运营），路由和操作按钮按角色控权（如"标记提现完成"客服和超级管理员可操作，"机器人配置""域名配置"仅超级管理员可操作）
- 群发消息系统：创建 broadcast_tasks，选择模板 + 筛选条件（活跃度层级/身份等级/自定义名单/全体）+ 定时发送时间；后台需实现一个发送队列消费者，按 Telegram 限速（约每秒30条到不同用户）逐条发送，更新 sent_count/click_count

## 3.7 提现申请管理

列表展示 withdraw_requests：状态（待处理/已完成）、申请积分数、申请时间、对应用户、风控标记（新账号大额提现）。管理员核实客服线下兑换完成后，点击"标记完成"：系统写入 points_ledger 兑换扣除记录（按 points_monthly_batches 的 FIFO 逻辑扣减），更新 users.points_balance，写入 withdraw_requests.completed_at / operator / status=已完成。

## 3.8 机器人配置

- 机器人列表（bots 表）：新增时输入 Token，保存前调用 Telegram `getMe` 接口校验有效性，成功则自动回填 bot_username，失败则提示错误不允许保存
- 当前生效标记：列表中可将某条记录设为"当前生效"（is_active=true），系统需保证同一时刻只有一条记录为 true（切换时自动将旧记录置为 false）
- 健康检测：见04文档定时任务，检测到当前生效机器人异常时，在此页面显著展示告警状态
- 切换后，短链跳转服务（见02文档"邀请"逻辑及04文档）读取的目标机器人应立即生效，无需重启服务（建议目标机器人配置做成可热更新的缓存或直接查库，不要硬编码在代码里）

## 3.9 域名配置

- 域名列表（domains 表）：新增域名、设置状态（启用/禁用/异常）
- 健康检测：见04文档定时任务，检测异常的域名自动置为"异常"状态并从短链生成的可分配池中排除
- 明确提示（UI文案层面）：域名被封后，此前已分发的历史短链无法通过后台恢复，此功能的意义在于降低单一域名故障的影响范围，而非补救已发生的故障
